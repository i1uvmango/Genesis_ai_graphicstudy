# Car_raywheel → genesis_vehicle SDK migration



## 1. API 정리

코드 convert 시 reference 용. 각 항목의 자세한 동작 / 적용 효과는 본문
migration 섹션에서 실제 사용될 때 설명.

### 핵심 class / function

| 이름 | 역할 |
|------|------|
| `VehiclePhysics(scene, car, sensor, cfg, n_envs)` | 5-step pipeline 실행 driver |
| `VehicleInputs(throttle, brake, steer)` | typed control input (MPPI 용) |
| `SkidSteerInputs`, `NoSteerInputs` | 차종별 input class |
| `add_vehicle(scene, URDF, preset)` | URDF entity + raycaster + cfg 생성 helper |
| `parse_urdf(URDF)` | URDF 에서 wheel 정보 추출 |
| `WheelRayPattern(positions)` | -Z 방향 ray pattern 정의 |
| `VehicleConfig`, `WheelConfig`, `ChassisConfig` | config object |

- **`VehiclePhysics`**: 매 step 5단계 (raycast → suspension → slip → tire → omega) 를 batched 로 돌리는 메인 엔진. 옛 `CarRayWheelPhysics` 와 동일 위치.
- **`VehicleInputs`**: `physics.step()` 에 넣을 input. positional args 대신 keyword 로 명확.
- **`SkidSteerInputs / NoSteerInputs`**: tank (좌/우 throttle 따로) / trailer 용. 
MPPI는 `VehicleInputs` 만 사용.
- **`add_vehicle`**: URDF 등록 + Raycaster 부착 + preset cfg 생성을 한 줄로. fine-grained 제어 필요하면 hand-wired (`parse_urdf` + `WheelRayPattern` 직접) 가능.
- **`parse_urdf`**: URDF parsing 해서 wheel position / radius / mass / joint name 추출.
- **`WheelRayPattern`**: 각 wheel 중심에서 -Z 방향 ray 1 개씩 정의하는 Genesis raycaster pattern.
- **`VehicleConfig / WheelConfig / ChassisConfig`**: vehicle / per-wheel / chassis parameter 보관 object. user override 가 URDF default 보다 우선.

### Car Setting options

| Axis | Options |
|----|------|
| **Steering** | `Ackermann`, `PartialAckermann`, `SkidSteer`, `NoSteer` |
| **Drivetrain** | `FWD`, `RWD`, `AWD`, `PerSide` |
| **Coupling** | `Independent`, `SameSideBelt` |
| **Tire model** | `PacejkaAnisotropic`, `CoulombIsotropic` |

**Steering**
- **`Ackermann`**: 좌우 front wheel 을 다른 각도로 (안쪽 더 sharp). 일반 passenger car 표준. **우리 케이스.**
- **`PartialAckermann`**: 100% Ackermann ↔ parallel steering 사이. 고속 차량 (F1 등) 에서 camber / tire 효과 보정.
- **`SkidSteer`**: front wheel steering 없음. 좌/우 throttle 차이로 회전. tank, 굴삭기, tractor.
- **`NoSteer`**: steering 자체 없음. trailer, 화물칸.

**Drivetrain** (전륜,후륜,사륜)
- **`FWD`** (Front-Wheel Drive): drive torque 가 front wheel 에만.
- **`RWD`** (Rear-Wheel Drive): rear wheel 에만. **우리 케이스 (Blender 차량).**
- **`AWD`** (All-Wheel Drive): 모든 wheel 에 분배.
- **`PerSide`**: 좌/우 따로 (tank 처럼 좌우 throttle input 이 따로일 때).

**Coupling** (탱크용)
- **`Independent`**: 각 wheel 이 독립적으로 회전. 일반 passenger car.
- **`SameSideBelt`**: 같은 쪽 wheel 들이 caterpillar belt 로 묶임 (tank track 처럼 한 쪽 wheel 들이 같이 도는 구조).

**Tire model**
- **`PacejkaAnisotropic`**: Magic formula. longitudinal / lateral 각각 비선형 곡선 + friction circle clamping. 자동차 sim 업계 표준.
- **`CoulombIsotropic`**: 단순 μN 마찰. isotropic (방향 무관). 옛 mesh-contact engine 의 기본 friction model. 빠르지만 부정확.

### Stability Hooks

| Hook | 발동조건 | 비고 |
|------|-------|---|
| `RollingResistance` | 항상 (tire force 후) | - |
| `LowSpeedRegularizer` | 저속 (pre-loop + post-tire), throttle/brake active 시 자동 OFF | `disable_when_control_active=True` default |
| `StaticFrictionLock` | brake>0.3 & V<0.5 | v_long만 &rarr; v_lat 추가 [slip_test](docs\tech\[26-05-23]_pacejka_slip_test.md) |

- **`RollingResistance`**: rolling resistance 추가. 차가 가속 / 감속 시 자연 감쇠 효과.
  → tire 가 굴러갈 때 변형 / 복원 과정에서 발생하는 작은 energy loss 로 차가 자연 감속.

- **`LowSpeedRegularizer`**: 저속 진동 (Pacejka 의 한계) 방지. `moving ∈ [0, 1]` factor 로 저속에서 tire force 끔 + omega 를 rolling-without-slip 쪽으로 끌어당김.
  → 다시 말해, 저속에서 Pacejka 가 수학적으로 깨지는 걸 막아 차량의 jitter 를 방지.

- **`StaticFrictionLock`** : 진짜 정지 (stick-slip). brake 누른 채로 저속이면 wheel 의 contact 위치를 기준점으로 잡고, 그 기준점 에 spring-damper force 적용, 경사로에서도 흘러내림 없이 정지함.(너무 크면 진동 나타남 &rarr; downscaling 후 사용)
  * 즉, `LowSpeedRegularizer` 에서 Pacejka 끄면 static friction 효과까지 같이 사라져 gravity 로 차가 미끄러지는데, 이것을 spring-damper force 를 추가로 인가해서 사라진 static friction 을 대체함.
  * SDK 기본 (k_spring=500K, k_damp=20K) 은 5t 트럭 기준 → **우리 코드는 mass-scaled 130K/5K 로 replace**.



### Sign convention (ISO 8855)

- `+X` forward, `+Y` left, `+Z` up
- `+throttle` 전진 가속, `brake ∈ [0, 1]`, `+steer` = right turn
- Genesis RHS / URDF axis 의 flip 은 SDK 내부에서 자동 처리

## 2. Override 항목

> SDK preset 의 default 를 그대로 쓰지 않고 우리 차량 / 시나리오에 맞게 override 3가지 적용. 모두 `★ OVERRIDE` 주석으로 코드에 명시되어 있음.

### 2.1 Drive torque: 1000 → 3000 N·m


```python
vcfg.drivetrain.t_drive_max = 3000.0   # 1_find_gt.py line 522
```

| 항목 | 값 |
|---|---|
| SDK 기본 (car_4w_rwd_ackermann) | 1000 N·m |
| 우리 override | **3000 N·m** |
| μ=1 마찰 한계 (rear axle 기준) | ≈ 6500 N (≈ 0.5g) |

**근거**: HillPeak ref CSV 가 등판 + 가속 동시 demand 시 1000 N·m 가 부족 &rarr; 3000 으로 올려 마찰원 직전까지 가용 토크 확보.

단점: 평지에선 마찰원을 약간 초과 → SDK 의 friction-circle clamp 가 자동으로 잘라줌. 마찰원 안에서 작동하므로 안전.

### 2.2 StaticFrictionLock: k_spring 500K → 130K (mass-scaled)

```python
# SDK 기본 (v0.5.8+ car preset 에 자동 포함):
#   StaticFrictionLock(brake_thr=0.3, v_thr=0.5, k_spring=500_000, k_damp=20_000)
# → 5t 트럭 기준 → 1.34t 우리 차에 stiff 함 → settling 진동/drift

# 1_find_gt.py line 530-541:
from genesis_vehicle import StaticFrictionLock
_sfl_idx = next((i for i, h in enumerate(vcfg.stability_hooks)
                 if isinstance(h, StaticFrictionLock)), None)
_sfl_new = StaticFrictionLock(brake_thr=0.3, v_thr=0.5,
                              k_spring=130_000.0, k_damp=5_000.0)
if _sfl_idx is None:
    vcfg.stability_hooks.append(_sfl_new)
else:
    vcfg.stability_hooks[_sfl_idx] = _sfl_new   # ★ replace 패턴
```

| 항목 | 값 |
|---|---|
| SDK 기본 k_spring / k_damp | 500_000 / 20_000 (5t 트럭) |
| 우리 차량 질량 | 1.34t |
| mass ratio | 1.34/5 = 0.268 |
| Override Hook: k_spring / k_damp | **130_000 / 5_000** (≈ SDK 기본 × 0.268) |

### 2.3 Mesh: convexify=False, decimate=False

```python
self.scene.add_entity(
    morph=gs.morphs.Mesh(
        file=cfg.hill_mesh,
        pos=(0.0, 0.0, -self.h_min),
        fixed=True,
        convexify=False,   # ★ OVERRIDE
        decimate=False,    # ★ OVERRIDE
    ),
)
```

**근거**: SDK / Genesis 기본은 `convexify=True, decimate=True` → mesh 가 **convex hull** 로 변환되어 raycaster 가 실제 도로 표면 대신 convex hull 표면에 hit.

문제: mesh를 blender와 동일하게 높이 조정하는 과정에서 collision 과 visual 이 달라졌음

| 측정 | 원본 mesh | Convex hull |
|---|---|---|
| HillRoad_P03 도로 face 경사 P95 | 7.2° | **40.4°** (가짜로 부풀려짐) |
| > 10° face 개수 | 3 / 1508 | 396 / 1508 |

* convex hull 표면이 실제 도로보다 위로 솟아 차가 시각적으로 떠있게 보이고, raycast 가 가짜 가파른 경사를 읽음.
* `convexify=False, decimate=False` 로 원본 OBJ 그대로 사용.
* 부작용: collision mesh face 수 증가 → BVH traversal 비용 증가 (`n_envs=2001` 에서 raycast 약간 느려질 수 있음). 정확도가 더 중요하므로 채택.
